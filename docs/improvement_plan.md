# ImmunoMatch Lightweight Workflow Improvement Plan

## Scope

This document proposes a modular upgrade path for the current lightweight ImmunoMatch reproduction project without modifying the existing workflow code. The goal is to add new modules only, so the current scripts remain intact and can still be used as the baseline reference.

The proposal is based on the published ImmunoMatch paper in *Nature Methods* and the public implementation boundary observed in the repository and packaged checkpoint behavior.

## Research Basis

The paper establishes the following public workflow facts:

- Positive examples are paired heavy/light chains observed in the same single B cell.
- Pseudo-negatives are generated by random shuffling of light-chain partners.
- A mixed ImmunoMatch model and chain-type-specific ImmunoMatch-κ / ImmunoMatch-λ models are used.
- The paper evaluates not only paired-cell discrimination, but also spatial VDJ reconstruction, donor-level B cell stage trends, cancer cohorts, and therapeutic antibody perturbations.
- The public package exposes inference, not a full training pipeline.

Reference: Guo et al., *Nature Methods* 23, 106-117 (2026). DOI: `10.1038/s41592-025-02913-x`.

## Design Goals

The improvement strategy should optimize four dimensions:

- Speed: reduce redundant checkpoint loads, repeated tokenization, and repeated pair scoring.
- Accuracy: improve data handling, calibration, and optional multi-signal scoring without retraining the original model.
- Functionality: expand from single CSV inference to reusable analysis workflows and multiple paper-style use cases.
- Extensibility: make the project easy to extend to new datasets, new batch scoring modes, and downstream analyses.

## Recommended Modular Upgrades

### 1. `immunomatch_cache`

Purpose: eliminate repeated expensive work.

Planned features:

- Model singleton cache for tokenizer and checkpoint objects.
- Pair-score memoization keyed by `(VH, VL, locus, model_name)`.
- On-disk cache for previously scored pairs.
- Hash-based deduplication before inference.

Expected benefit:

- Faster repeated scoring across multiple files.
- Lower CPU and disk overhead.
- Better performance for figure workflows where the same pairs are reused repeatedly.

### 2. `immunomatch_batch_engine`

Purpose: improve inference throughput for large CSVs or all-by-all candidate matrices.

Planned features:

- Chunked batch scheduling.
- Adaptive batch sizing based on GPU/CPU availability.
- Streamed CSV reading instead of full-memory loading.
- Parallel preprocessing with separate tokenization workers.

Expected benefit:

- Better throughput on large datasets.
- Lower memory usage.
- Easier scaling from toy data to donor-level or tissue-level data.

### 3. `immunomatch_pair_builder`

Purpose: standardize how positives and pseudo-negatives are built.

Planned features:

- Deterministic pseudo-negative generation with seed control.
- Stratified shuffling by locus, sample, donor, or cell type.
- Duplicate and collision checks to avoid accidentally reusing the observed positive pair.
- Optional hard-negative construction by sampling similar chains rather than only random shuffles.

Expected benefit:

- Better evaluation stability.
- More reproducible pseudo-negative construction.
- A path toward more challenging benchmarks without changing the original model.

### 4. `immunomatch_metrics`

Purpose: add richer evaluation and calibration.

Planned features:

- AUC-ROC, PR-AUC, accuracy, F1, specificity, sensitivity, MCC.
- Threshold sweep and best-threshold selection.
- Score calibration curves and reliability diagrams.
- Bootstrap confidence intervals for scores and metrics.
- Per-locus and per-dataset summary tables.

Expected benefit:

- More informative reporting than a single score cutoff.
- Better comparison with the manuscript and downstream datasets.
- Improved decision support for practical screening.

### 5. `immunomatch_visuals`

Purpose: generate publication-style outputs from the lightweight workflow.

Planned features:

- Score distributions for positive vs pseudo-negative pairs.
- ROC and PR curves.
- Confusion matrix heatmaps.
- Donor-level or sample-level score summaries.
- Consistent SVG/PDF export helpers.

Expected benefit:

- Faster reuse for slides, notes, and GitHub README previews.
- A consistent visual language across future datasets.

### 6. `immunomatch_dataset_adapters`

Purpose: make the project extensible to more paper-style inputs.

Planned features:

- Adapter for single-cell paired CSVs.
- Adapter for spatial VDJ reconstruction inputs.
- Adapter for therapeutic antibody inputs.
- Adapter for cancer cohort summary tables.
- Common schema validation and column normalization.

Expected benefit:

- One project can support several manuscript-style workflows.
- Easier integration of new public datasets.
- Less duplicated parsing code.

### 7. `immunomatch_analysis_recipes`

Purpose: package paper-style analyses as reusable recipes.

Planned recipes:

- `tonsil_stage_analysis`
- `kappa_lambda_cross_test`
- `spatial_repair_comparison`
- `therapeutic_antibody_perturbation`
- `cancer_cohort_summary`

Expected benefit:

- The project can move beyond a single toy example.
- Clear mapping from the paper's figures to executable workflows.
- Easier future expansion without rewriting the core scorer.

### 8. `immunomatch_cli`

Purpose: provide a clean command-line interface while keeping the current scripts untouched.

Planned commands:

- `download-models`
- `score-pairs`
- `build-pseudo-negatives`
- `run-lightweight-workflow`
- `summarize-results`

Expected benefit:

- Easier user adoption.
- More consistent argument parsing.
- Clear separation between public entry points and internal helpers.

## Accuracy-Oriented Enhancements

These improvements do not require retraining the current public checkpoints.

### A. Ensembling by viewpoint

Use a lightweight ensemble of:

- raw ImmunoMatch score
- calibrated score
- optional score from a sequence-only heuristic baseline

This can improve stability on ambiguous pairs and make the output more robust.

### B. Confidence-aware filtering

For large candidate sets, only surface top-k pairs per heavy chain or per cell after ranking, with a confidence threshold and optional margin against the runner-up.

### C. Hard-negative support

When the project is later extended beyond toy evaluation, add optional hard negatives sampled from similar loci or same donor groups. This can provide a more realistic stress test than pure random shuffles.

### D. Calibration layer

Add a post-hoc calibration module to map raw scores to better interpretable probabilities or risk scores.

## Speed-Oriented Enhancements

Recommended priority order:

1. Model/tokenizer cache.
2. Score memoization.
3. Chunked batch inference.
4. Streaming input reader.
5. Parallel preprocessing.
6. Optional GPU-aware batch sizing.

These changes can be added without touching the existing baseline scripts by placing them in new modules and using new entry points.

## Extensibility Roadmap

### Phase 1: Baseline optimization

- Add cache and batch engine.
- Add schema validation and a CLI.
- Keep output format identical to the current workflow.

### Phase 2: Paper-style analysis expansion

- Add the analysis recipes for tonsil, spatial VDJ, cancer, and therapeutic antibody cases.
- Add richer plots and metric tables.

### Phase 3: Advanced scoring stack

- Add calibration and optional ensemble scoring.
- Add hard-negative support and confidence-aware ranking.

### Phase 4: External dataset plug-in system

- Add adapters for new paired antibody datasets.
- Support user-defined column mapping and dataset-specific preprocessing.

## Proposed File Layout For Future Modules

```text
immunomatch_ext/
├── __init__.py
├── cache.py
├── batch_engine.py
├── pair_builder.py
├── metrics.py
├── visuals.py
├── dataset_adapters.py
├── analysis_recipes.py
├── cli.py
└── calibration.py
```

This layout keeps the current code untouched while making future expansion explicit and maintainable.

## Recommended Order Of Execution Later

When you approve implementation, the most practical order is:

1. Add `immunomatch_cache` and `immunomatch_batch_engine` for immediate speed gains.
2. Add `immunomatch_metrics` and `immunomatch_visuals` for better reporting.
3. Add `immunomatch_dataset_adapters` and `immunomatch_analysis_recipes` for paper-style extensibility.
4. Add `immunomatch_cli` to expose the new modules cleanly.
5. Add calibration and optional ensemble logic last.

## Current Recommendation

For the existing lightweight reproduction, the highest-value next step is a cache-aware batch engine with a unified CLI wrapper. That gives the largest speed gain with the smallest disruption and creates a clean base for later accuracy and analysis upgrades.
